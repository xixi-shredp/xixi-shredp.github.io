OOO CPU Issue Stage

Posted on 2025-10-29 In Arch , ooo , issue
乱序执行 CPU 发射阶段简介
  • InOrder Issue Logic
    • 指令需要等待所有先前的指令都发射后才可以发射
    • 实现简单：仅需要 scoreboarding
      • a data dependence table: 每个 entry 代表一个寄存器是否 available
      • a resource table: 追踪所有功能单元的状态
    • 特殊的顺序处理器： VLIW processors
  • Out-Of-Order Issue Logic
    • 往往是处理器流水线的关键路径

发射阶段 (Issue Stage) 的任务：接收来自重命名之后的指令（顺序），乱序地送到功能单元去执行

主要电路：

  1. 发射队列 (Issue Queue) / 保留站 (Reservation Station)
  2. 分配 (Allocation) 电路
  3. 选择 (Select) 电路 / 仲裁 (Arbiter) 电路
  4. 唤醒 (Wake-up) 电路

流水线

发射队列中的指令被仲裁电路选中执行的条件：

  1. 源操作数均已准备好
  2. 仲裁电路允许发射
  3. 可以从寄存器堆 / payload RAM / 旁路网络中获得源操作数的值

发射阶段可以划分为唤醒和仲裁两个阶段

非数据捕捉结构的流水线

方案1: 指令执行完后对其他指令进行唤醒 (Tomasulo 算法)

  • 优势：设计简单
  • 缺陷：两条 RAW 相关的指令不能背靠背执行，中间相差 3 个周期

方案2: 通过广播将唤醒阶段提前 (背靠背执行)

为保证背靠背执行，必须保证前一条指令的 Select 阶段和后一条指令的 Wake-up 阶段之间原子化串行

由于 Select-Wakeup 阶段的延时较高，可以将其再分成 2 个周期：


但这样无法保证上述的原子性，所以两条 RAW 相关的指令无法背靠背执行，会相差一个周期

对于多周期指令，还需要在词基础上采用延时广播/延时唤醒, 推测唤醒等策略保证正确性
