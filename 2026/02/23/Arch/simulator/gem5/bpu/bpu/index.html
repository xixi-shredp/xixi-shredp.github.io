Gem5 BDP 实现分析

Created2026-02-23|Updated2026-04-13|Archsimulatorgem5bpu
|Post Views:

Gem5 version: v25.1.0.0

本文基于当前仓库 src/cpu/pred 下的实现与 src/cpu/pred/BranchPredictor.py 中的参数定义，聚焦 gem5 里已经提供的方向预测器。这里的“方向预测器”指条件分支 taken / not-taken 的预测逻辑；BTB、RAS、Indirect predictor 不作为本文主体，但与部分组合预测器会发生协同。

Overview

当前 Gem5 中可直接实例化的方向预测器:

  • LocalBP
  • TournamentBP
  • BiModeBP
  • TAGE
  • LTAGE
  • TAGE_SC_L_64KB
  • TAGE_SC_L_8KB
  • MultiperspectivePerceptron8KB
  • MultiperspectivePerceptron64KB
  • MultiperspectivePerceptronTAGE8KB
  • MultiperspectivePerceptronTAGE64KB
  • GshareBP

另外还有若干抽象或子组件类：

  • ConditionalPredictor：条件预测器基类
  • TAGEBase / TAGE_SC_L_TAGE / MPP_TAGE：TAGE 家族核心
  • LoopPredictor / TAGE_SC_L_LoopPredictor / MPP_LoopPredictor
  • StatisticalCorrector / TAGE_SC_L_*_StatisticalCorrector / MPP_StatisticalCorrector*
  • MultiperspectivePerceptron / MultiperspectivePerceptronTAGE：感知机系基类

共通参数与总线入口

大部分预测器都通过 BranchPredictor.py 注入到 CPU 配置里。共通参数有：

  • instShiftAmt
    • 作用：索引前先右移 PC。对定长 ISA（Arm/RISC-V）通常设为 2，对 x86 常设 0。
    • 影响：几乎所有索引函数都会用到它。设错会导致别名冲突或有效位浪费。
  • speculativeHistUpdate / speculative_update
    • 作用：是否在预测时先投机更新历史，并在 squash 时恢复。
    • 影响：前端更激进时通常有利于精度，但需要正确恢复逻辑。
    • 注意：MultiperspectivePerceptron 明确 fatal_if(speculative_update)，当前实现不支持。
  • takenOnlyHistory
    • 作用：TAGE 家族是否只记录 taken 分支到全局历史。
    • 影响：更接近部分现代服务器 CPU；适合降低 not-taken 噪声，但会改变索引分布。

各预测器实现分析

LocalBP: 饱和计数器实现的局部历史表

相关文件：

  • src/cpu/pred/2bit_local.cc
  • src/cpu/pred/2bit_local.hh
  • src/cpu/pred/BranchPredictor.py

预测方式：

  • 只使用 PC 索引，不使用全局历史。
  • 索引为 (branch_addr >> instShiftAmt) & indexMask。
  • 表项是饱和计数器，取最高位作为预测结果。

更新方式：

  • lookup() 只读计数器。
  • update() 在提交路径上按真实结果对对应计数器自增或自减。
  • squash() 和 updateHistories() 基本为空，因为没有可恢复历史。

可修改参数：

  • localPredictorSize
  • localCtrBits
  • 继承自父类的 instShiftAmt

实现约束和注意点：

  • 构造函数里 localPredictorSets = localPredictorSize / localCtrBits，并要求 localPredictorSize 与 localPredictorSets 都是 2 的幂。
  • 这意味着这里的 localPredictorSize 不是“表项个数”的直观语义，而更像“总预算字段”，调参时必须保证 localPredictorSize / localCtrBits 仍为 2 的幂。

调参建议：

  • 想降低别名冲突：优先增大 localPredictorSize。
  • 想增强单分支稳定性：适当增大 localCtrBits，但不要过大，否则训练变慢。
  • 适合强局部性、弱全局相关 workload；对跨分支相关性强的程序提升有限。

TournamentBP: 锦标赛预测器，局部历史表和全局历史表通过一个选择预测表选择

相关文件：

  • src/cpu/pred/tournament.cc
  • src/cpu/pred/tournament.hh
  • src/cpu/pred/BranchPredictor.py

预测方式：

  • 同时计算三部分：
    • local predictor：PC 先索引 local history table，再用 local history 索引 local counter table。
    • global predictor：用全局历史索引 global counter table。
    • choice predictor：也用全局历史索引，决定选 local 还是 global。
  • 最终结果由 choiceCtrs 决定采用 local 还是 global 预测。

更新方式：

  • updateHistories() 会先投机更新全局历史；条件分支还会更新 local history。
  • squash() 会恢复全局历史和 local history。
  • update() 在提交时：
    • 更新 choice predictor，只在 local/global 不同且存在有效 local 预测时训练。
    • 更新 global counter。
    • 更新 local counter。

可修改参数：

  • localPredictorSize
  • localCtrBits
  • localHistoryTableSize
  • globalPredictorSize
  • globalCtrBits
  • choicePredictorSize
  • choiceCtrBits
  • instShiftAmt
  • speculativeHistUpdate

实现约束和注意点：

  • localPredictorSize、localHistoryTableSize、globalPredictorSize、choicePredictorSize 都要求是 2 的幂。
  • 全局历史位数取 max(log2(globalPredictorSize), log2(choicePredictorSize))。
  • local history 的位数由 ceilLog2(localPredictorSize) 推出。

调参建议：

  • local 强、global 弱的 workload：增大 localHistoryTableSize 与 localPredictorSize。
  • 分支间相关强：增大 globalPredictorSize 和 choicePredictorSize。
  • choicePredictorSize 太小会让选择器饱和过快，local/global 优势切换不稳。
  • 默认是经典折中方案，适合作为基线，但一般会被 TAGE 系超过。

BiModeBP

相关文件：

  • src/cpu/pred/bi_mode.cc
  • src/cpu/pred/bi_mode.hh
  • src/cpu/pred/BranchPredictor.py

预测方式：

  • 先用 PC 索引 choiceCounters，判断该分支是“偏向 taken”还是“偏向 not taken”。
  • 再用 (PC ^ GHR) 索引两个方向数组：
    • takenCounters
    • notTakenCounters
  • choice predictor 决定最终使用哪个方向数组的结果。

更新方式：

  • updateHistories() 只投机更新全局历史。
  • squash() 恢复全局历史。
  • update() 只更新被选中的方向数组；未被选中的方向数组不训练。
  • choice predictor 的更新策略比较保守：
    • 若最终预测正确，只有当选择器方向也和正确结果一致时才训练 choice。
    • 若最终预测错误，则总是训练 choice。

可修改参数：

  • globalPredictorSize
  • globalCtrBits
  • choicePredictorSize
  • choiceCtrBits
  • instShiftAmt
  • speculativeHistUpdate

实现约束和注意点：

  • choicePredictorSize 和 globalPredictorSize 必须为 2 的幂。
  • Bi-Mode 的核心目标不是提高“复杂相关性建模”，而是降低不同偏置分支之间的负干扰。

调参建议：

  • 若 workload 里大量分支具有稳定偏置，但共享全局表后干扰严重，Bi-Mode 通常比纯 gshare 更稳。
  • 优先先调大 choicePredictorSize 以减少分支偏置混淆，再调 globalPredictorSize。

GshareBP

相关文件：

  • src/cpu/pred/gshare.cc
  • src/cpu/pred/gshare.hh
  • src/cpu/pred/BranchPredictor.py

预测方式：

  • 经典 gshare：((PC >> instShiftAmt) ^ globalHistoryReg) & globalHistoryMask
  • 结果来自单张全局计数器表。

更新方式：

  • updateHistories() 投机更新全局历史。
  • squash() 恢复全局历史。
  • update() 在提交时按真实结果更新对应饱和计数器。

可修改参数：

  • global_predictor_size
  • global_counter_bits
  • instShiftAmt
  • numThreads

实现约束和注意点：

  • global_predictor_size 必须为 2 的幂。
  • 这个实现直接继承 BPredUnit，不是 ConditionalPredictor，但功能上就是方向预测器。

调参建议：

  • 适合做基线或教学用途。
  • 优先增大 global_predictor_size；global_counter_bits 维持 2 或 3 通常已经够用。
  • 若效果不够，一般直接切到 BiModeBP、TournamentBP 或 TAGE 系更划算。

TAGE

相关文件：

  • src/cpu/pred/tage.cc
  • src/cpu/pred/tage.hh
  • src/cpu/pred/tage_base.cc
  • src/cpu/pred/tage_base.hh
  • src/cpu/pred/BranchPredictor.py

预测方式：

  • 基础 bimodal 作为后备预测
  • 多个历史长度递增的 tagged table 共同组成 TAGE
  • tagePredict() 会：
    • 计算所有表的 index 和 partial tag。
    • 找到最长匹配表 hitBank。
    • 找到次长匹配表 altBank。
    • 对“新分配且弱计数器”的项，可能退回使用 alternate prediction。
  • 最终来源可能是：
    • 最长匹配表
    • 次长匹配表
    • bimodal

更新方式：

  • condBranchUpdate() 仅在预测错误且还有更长历史表可用时尝试分配新项。
  • handleTAGEUpdate() 更新 provider 项；若 provider 的 u 位为 0，还会顺带训练 alt / base。
  • u（useful）位定期衰减，周期由 logUResetPeriod 控制。
  • updateHistories() 负责 path history 和 global history 的更新；支持投机和恢复。
  • squash() 会恢复历史并重算 folded history。

可修改参数：

  • nHistoryTables
  • minHist
  • maxHist
  • tagTableTagWidths
  • logTagTableSizes
  • logRatioBiModalHystEntries
  • tagTableCounterBits
  • tagTableUBits
  • histBufferSize
  • pathHistBits
  • logUResetPeriod
  • numUseAltOnNa
  • initialTCounterValue
  • useAltOnNaBits
  • maxNumAlloc
  • noSkip
  • speculativeHistUpdate
  • takenOnlyHistory
  • instShiftAmt

实现约束和注意点：

  • tagTableTagWidths、logTagTableSizes 长度必须为 nHistoryTables + 1，其中第 0 项对应 bimodal。
  • tagTableTagWidths[0] 必须为 0。
  • histBufferSize 要明显大于 maxHist * 3。
  • tagTableUBits 当前要求为 1 或 2。

调参建议：

  • 追求精度时，优先扩大 maxHist 与表容量；但 tag 太短会导致冲突上升。
  • pathHistBits 增大可改善 path-sensitive workload，但过大可能增加噪声。
  • maxNumAlloc 越大，学习更快，但也更容易污染。
  • takenOnlyHistory=True 可作为现代服务器前端风格的实验开关。

LTAGE

相关文件：

  • src/cpu/pred/ltage.cc
  • src/cpu/pred/ltage.hh
  • src/cpu/pred/loop_predictor.cc
  • src/cpu/pred/loop_predictor.hh
  • src/cpu/pred/BranchPredictor.py

预测方式：

  • 先调用 TAGE 得到基准预测。
  • 再调用 loop predictor：
    • 如果识别到循环分支并达到高置信度，就覆盖 TAGE 的结果。
  • provider 会标记最终来自 TAGE 还是 Loop predictor。

更新方式：

  • 提交时同时更新：
    • TAGE 统计与表项
    • Loop predictor 的统计、置信度、迭代次数、年龄
  • 若投机历史开启，squash 时既恢复 TAGE 也恢复 loop predictor 的投机状态。

可修改参数：

  • 所有 TAGE 参数
  • loop_predictor 子对象参数：
    • logSizeLoopPred
    • withLoopBits
    • loopTableAgeBits
    • loopTableConfidenceBits
    • loopTableTagBits
    • loopTableIterBits
    • logLoopTableAssoc
    • useSpeculation
    • useHashing
    • useDirectionBit
    • restrictAllocation
    • initialLoopIter
    • initialLoopAge
    • optionalAgeReset

Loop predictor 的实现要点：

  • 通过 tag 和 set/way 查表识别循环。
  • currentIter 与 numIter 配合判断“下一次是否退出循环”。
  • 只有高置信度时才接管最终预测。
  • 预测正确可提升 age / confidence，模式失配会释放表项。

调参建议：

  • 循环密集 workload：增大 logSizeLoopPred、loopTableIterBits。
  • 循环退出模式复杂：启用 useDirectionBit 往往更稳。
  • 如果误分配多，可打开 restrictAllocation 或提高 loopTableConfidenceBits。
  • 若并非循环主导程序，LTAGE 相比纯 TAGE 的增益未必明显。

TAGE_SC_L_8KB / TAGE_SC_L_64KB

相关文件：

  • src/cpu/pred/tage_sc_l.cc
  • src/cpu/pred/tage_sc_l.hh
  • src/cpu/pred/tage_sc_l_8KB.hh/.cc
  • src/cpu/pred/tage_sc_l_64KB.hh/.cc
  • src/cpu/pred/statistical_corrector.cc
  • src/cpu/pred/BranchPredictor.py

预测方式：

  • 结构为 TAGE + Loop Predictor + Statistical Corrector。
  • 预测顺序：
    • 先跑 TAGE。
    • 再让 loop predictor 视置信度决定是否覆盖。
    • 最后让 SC 基于偏置项、GEHL 历史项、置信度门控决定是否修正最终结果。

更新方式：

  • 提交时依次更新：
    • TAGE
    • Loop predictor
    • Statistical corrector
  • updateHistories() 同时维护 TAGE history 与 SC history。
  • squash() 时除了 LTAGE 自己的恢复，还会恢复 SC 的历史状态。

TAGE_SC_L 特有实现点：

  • TAGE_SC_L_TAGE 使用成对 history length 和共享物理表的技巧，减少位预算。
  • 短 tag 与长 tag 的表分别按 factor 映射，具有“部分相联”的效果。
  • getUseAltIdx() 不是固定 0，而是结合 bank 和置信度选择不同计数器。

SC 的实现要点：

  • 先算 bias 项，再叠加多个 GEHL 预测和。
  • 通过 highConf / medConf / lowConf 与 |lsum| 门控，决定是否推翻 TAGE/Loop 的结果。
  • 提交时更新 bias 表、GEHL 权重、阈值计数器 pUpdateThreshold。

8KB 与 64KB 的区别：

  • TAGE history 表规模、history 深度、skip 模式不同。
  • LoopPredictor 的 logSizeLoopPred 不同。
  • StatisticalCorrector 使用的 GEHL 组数、历史长度、局部历史项数不同。
  • 64KB 版本更深、更宽，适合追求精度；8KB 版本更强调面积约束。

主要可修改参数：

  • 所有 TAGE_SC_L_TAGE 参数
  • 所有 LoopPredictor 参数
  • 所有 StatisticalCorrector 参数
  • 8KB / 64KB 版本中的具体 GEHL 长度、bias 大小、局部历史项数

调参建议：

  • 预算严格受限：从 TAGE_SC_L_8KB 开始。
  • 追求精度：优先用 TAGE_SC_L_64KB。
  • 若 SC 经常过度纠正：
    • 降低 logBias、减小 GEHL 数量或历史长度。
    • 提高 chooserConfWidth / updateThresholdWidth，让 SC 更保守。
  • 若长相关性不足：
    • 提高 maxHist，增加 GEHL 历史长度向量。
  • 如果恢复开销或实现复杂度是重点，可先固定 speculativeHistUpdate=False。

MultiperspectivePerceptron8KB / 64KB

相关文件：

  • src/cpu/pred/multiperspective_perceptron.cc
  • src/cpu/pred/multiperspective_perceptron.hh
  • src/cpu/pred/multiperspective_perceptron_8KB.hh/.cc
  • src/cpu/pred/multiperspective_perceptron_64KB.hh/.cc
  • src/cpu/pred/BranchPredictor.py

预测方式：

  • 不是单一全局历史 perceptron，而是“多视角”特征融合。
  • 特征来源包括：
    • global history
    • path history
    • local history
    • IMLI 风格历史
    • modulo history / modulo path
    • acyclic history
    • recency stack
    • blurry path 等
  • computeOutput() 汇总各特征表的带符号权重和，得到 yout。
  • 当 |yout| <= threshold 时，会退化到 bestval / 低置信度策略。
  • 可选 filter table 对“始终 taken / 始终 not-taken”的分支直接短路。

更新方式：

  • 当前实现不支持 speculative update。
  • 提交时：
    • 先处理 filter table 的状态迁移。
    • 若需要训练，则调用 train() 调整各特征表权重和符号位。
    • 再更新多种历史结构。
  • record_mask 控制某些“已过滤分支”是否仍写入历史。

可修改参数：

  • num_filter_entries
  • num_local_histories
  • local_history_length
  • block_size
  • pcshift
  • threshold
  • bias0
  • bias1
  • biasmostly0
  • biasmostly1
  • nbest
  • tunebits
  • hshift
  • imli_mask1
  • imli_mask4
  • recencypos_mask
  • fudge
  • n_sign_bits
  • pcbit
  • decay
  • record_mask
  • hash_taken
  • tuneonly
  • extra_rounds
  • speed
  • initial_theta
  • budgetbits
  • initial_ghist_length
  • ignore_path_size

8KB / 64KB 默认差异：

  • budgetbits
  • num_local_histories
  • num_filter_entries
  • imli_mask1
  • imli_mask4
  • recencypos_mask

调参建议：

  • 想提高感知机表达能力：增大 budgetbits、num_local_histories、相关 feature mask。
  • 想减少训练震荡：
    • 降低 extra_rounds
    • 调整 speed
    • 增大 initial_theta
  • 若 filter 误杀可学习分支：
    • 减少 num_filter_entries
    • 关闭或减弱 decay
  • 若 path 特征收益不高，可设 ignore_path_size=True 先把预算转移给别的结构。

MultiperspectivePerceptronTAGE8KB / 64KB

相关文件：

  • src/cpu/pred/multiperspective_perceptron_tage.cc
  • src/cpu/pred/multiperspective_perceptron_tage.hh
  • src/cpu/pred/multiperspective_perceptron_tage_8KB.hh/.cc
  • src/cpu/pred/multiperspective_perceptron_tage_64KB.hh/.cc
  • src/cpu/pred/tage_base.cc
  • src/cpu/pred/statistical_corrector.cc
  • src/cpu/pred/loop_predictor.cc
  • src/cpu/pred/BranchPredictor.py

预测方式：

  • 结构为 MPP-TAGE core + Loop predictor + SC + perceptron partial sum。
  • 流程：
    • 先用 MPP_TAGE::tagePredict() 预测。
    • 再用 loop predictor 尝试覆盖。
    • 再把 perceptron partial sum 加到初始分数上，交给 SC 统一裁决。
  • 与普通 TAGE_SC_L 相比，它把一部分判别能力交给 perceptron 特征。

更新方式：

  • 提交时更新：
    • Loop predictor
    • perceptron partial tables
    • SC
    • MPP_TAGE
    • 自己维护的 recency / acyclic / modulo / blurry 历史
  • MPP_TAGE 自己也改写了若干 TAGE 行为：
    • adjustAlloc() 对“预测已正确”的情况降低分配概率。
    • handleAllocAndUReset() 和 resetUctr() 的策略不同于基础 TAGE。
    • updateHistories() 使用不同的 path/global history 编码方式。

可修改参数：

  • 所有 MultiperspectivePerceptron 基类参数
  • 所有 MPP_TAGE 参数
  • 所有 MPP_LoopPredictor 参数
  • 所有 MPP_StatisticalCorrector* 参数

调参建议：

  • 当纯 TAGE_SC_L 仍不足以捕获复杂长相关、跨历史形态相关时，优先试这个系列。
  • 64KB 版适合追求高精度；8KB 版适合预算受限实验。
  • 若行为过于激进，先降低 perceptron 相关训练强度，再调 SC 阈值；不要一开始同时大改 TAGE 与感知机两部分。

组合子组件参数修改

LoopPredictor 参数指南

  • logSizeLoopPred
    • 增大表容量，降低循环分支冲突。
  • loopTableIterBits
    • 增大可表示的最大迭代次数。
  • loopTableConfidenceBits
    • 增大后更难接管预测，但更稳。
  • useHashing
    • 开启后索引分布更均匀，通常建议保留。
  • useDirectionBit
    • 对非“多数 taken、末次 not-taken”的循环很有帮助。
  • restrictAllocation
    • 开启后更保守，降低污染。

StatisticalCorrector 参数指南

  • logBias
    • bias 表大小。增大有利于区分不同分支上下文。
  • gnb/pnb/snb/tnb/bwnb/inb
    • GEHL 组数。
  • gm/pm/sm/tm/bwm/im
    • 每组 GEHL 的历史长度向量。
  • logGnb/logPnb/logSnb/logTnb/logBwnb/logInb
    • GEHL 表项数。
  • chooserConfWidth
    • 决定何时相信 SC 覆盖主预测器。
  • updateThresholdWidth、pUpdateThresholdWidth
    • 决定训练阈值和训练激进程度。
  • scCountersWidth
    • 单个 SC 计数器的动态范围。

调参原则：

  • 误纠正多：提高阈值宽度，减少 GEHL 长度或组数。
  • 长历史收益不足：增加较长历史项，但要同步扩大表大小，否则冲突会上升。
  • 局部模式强：优先增加 local history 相关 GEHL，而不是盲目增加全局历史。

实际调参建议

先选系列，再细调参数

  • 教学基线：GshareBP、LocalBP
  • 传统中档基线：TournamentBP、BiModeBP
  • 通用高精度：TAGE
  • 循环多：LTAGE
  • 高精度主力：TAGE_SC_L_64KB
  • 面积受限高精度：TAGE_SC_L_8KB
  • 研究复杂长相关：MultiperspectivePerceptron*
  • 同时要 TAGE 稳定性和感知机表达力：MultiperspectivePerceptronTAGE*

不同目标下的优先调参顺序

目标是减少 alias / 冲突：

  • 增大表大小参数
  • 增大 tag 宽度
  • 适当调整 instShiftAmt

目标是学习更长相关：

  • 增大 maxHist
  • 增加 history tables 数量
  • 拉长 SC / perceptron 的历史长度向量

目标是降低训练噪声：

  • 增大计数器位宽
  • 提高 SC 阈值
  • 降低分配积极性，如减小 maxNumAlloc

目标是减小错误恢复复杂度：

  • 关闭或限制 speculative history update
  • 选择非投机式实现更清晰的系列，例如 MPP 当前天然非投机

一个实用的调参流程

  1. 先固定 instShiftAmt 与前端 ISA 一致。
  2. 先选一个系列，不要跨系列同时大改。
  3. 先扩容量，再改历史长度，再改训练门限。
  4. 每次只改 1 到 3 个核心参数。
  5. 记录 MPKI、IPC、预测准确率、恢复次数，而不是只看单一准确率。
